# خادم Railway لتيسير — بنية مفصولة (واتساب ⇄ Supabase ⇄ ذكاء Groq)

## الفكرة الجوهرية

النظام مقسوم إلى وحدتين **منفصلتين تماماً**، لا يعرف أحدهما الآخر، ويتواصلان فقط
عبر Supabase، ومع التشغيل الافتراضي كخدمة واحدة توجد إشارات داخلية فورية تختصر انتظار polling. هذا يجعل النظام سريعاً جداً وموثوقاً: لو تعطّل الذكاء، تبقى
الرسائل محفوظة؛ ولو تعطّل الإرسال، لا يضيع أي رد.

جلسة واتساب لا تُحفظ في Supabase نهائياً في هذه النسخة. يتم استخدام جلسة محلية
داخل تشغيل Railway الحالي فقط، حتى لا يحدث تعارض جلسات قديم يسبب تسجيل خروج مفاجئ.
بعد Redeploy أو حذف ملفات البيئة سيظهر QR جديد وهذا مقصود.

```
                    ┌──────────────────────────────┐
   رسالة واتساب  →  │  server.js  (خادم واتساب)      │
                    │  • يستقبل الرسالة              │
                    │  • يحفظها فوراً في Supabase     │ ── (لا يعرف Groq إطلاقاً)
                    │  • يراقب outbox ويرسل أي رد    │
                    └──────────────┬───────────────┘
                                   │  Supabase
             aiQueue (وارد) ↓      │      ↑ outbox (صادر)
                    ┌──────────────┴───────────────┐
                    │  ai-worker.js (عامل الذكاء)   │
                    │  • يقرأ الرسالة من aiQueue     │ ── (لا يعرف واتساب إطلاقاً)
                    │  • يبحث في المعرفة/الأسئلة     │
                    │  • إن لم يجد → يرد عبر Groq    │
                    │  • يكتب الرد في outbox         │
                    └──────────────────────────────┘
```

### تدفّق رسالة واحدة (خطوة بخطوة)
1. عميل يرسل «مرحبا» إلى رقم البوت.
2. `server.js` يستقبلها ويحفظها فوراً في:
   `stores/{storeId}/bots/{botId}/conversations/{phone}/messages`
   ويضيف مهمة في `.../aiQueue` بحالة `pending`.
3. `ai-worker.js` يستقبل إشارة داخلية فورية عند إنشاء `aiQueue`، ومعها polling سريع كشبكة أمان، ثم يقرأ الرسالة:
   - يبحث في **الردود الجاهزة** و**الأسئلة الشائعة** → إن وُجد تطابق، يرد فوراً بدون Groq.
   - وإلا يستدعي **Groq** بمفتاح البوت الخاص مع قاعدة المعرفة والقوانين كسياق.
4. يكتب الرد في المحادثة ويضيفه إلى `.../outbox` بحالة `pending`.
5. `server.js` يستقبل إشارة داخلية فورية عند إنشاء `outbox`، يلتقط الرد ويرسله للرقم، ثم يعلّمه `sent`.

الخط السريع يعمل داخل نفس عملية Railway، ومعه مراقب احتياطي سريع. إعدادات الذكاء تُحفظ في الذاكرة وتُحدّث دورياً بالخلفية، لذلك لا يتأخر كل رد بسبب قراءة الإعدادات من قاعدة البيانات، ولو حصل ضغط مؤقت لا يفقد البوت مفتاح Groq أو سياقه ولا يتوقف فوراً.

## ملفات المستودع

| الملف | الدور |
|---|---|
| `index.js` | مُشغّل واحد يشغّل الوحدتين معاً (أمر البدء الافتراضي). |
| `server.js` | خادم واتساب: استقبال + حفظ + إرسال. **بدون أي ذكاء.** |
| `ai-worker.js` | عامل Groq: قراءة + تحليل + رد. **بدون أي واتساب.** |
| `firestore-compat.js` | طبقة توافق تجعل الكود القديم يقرأ/يكتب في Supabase فقط. |
| `firestore-writer.js` | طبقة البيانات المشتركة (كل القراءة/الكتابة في Supabase). |
| `firestore-session-store.js` | أداة تنظيف للجلسات القديمة فقط؛ لا يتم استخدامها لحفظ جلسة جديدة. |
| `package.json` / `Procfile` | إعداد النشر. |

## خطوات النشر على Railway (5 دقائق)

### 1) تأكد من حذف متغيرات النظام السابق
احذف من Railway أي متغير قديم خاص بالنظام السابق، واستخدم المتغيرات أدناه فقط.

### 2) المتغيرات في Railway → Variables
| المتغير | القيمة |
|---|---|
| `APP_BACKEND_URL` | رابط الـ backend كما يظهر في لوحة تيسير |
| `APP_BACKEND_PUBLISHABLE_KEY` | مفتاح النشر العام كما يظهر في لوحة تيسير — لا تحتاج مفتاحاً سرياً |
| `TAYSIR_STORE_ID` | id المتجر (`stores/xxxx`) |
| `TAYSIR_BOT_ID` | id البوت (`stores/xxx/bots/yyy`) |
| `SERVICE_TOKEN` | إجباري — نفس قيمة `railwayApiKey` في إعدادات البوت |
| `FAST_QUEUE_MODE` | (اختياري) افتراضي true — يفرض فحصاً سريعاً للطوابير حتى لو بقيت إعدادات قديمة في قاعدة البيانات |
| `AI_BURST_WAIT_MS` | (اختياري) افتراضي 0ms — اتركه 0 حتى يرد على كل رسالة؛ يمكن رفعه قليلاً فقط إذا أردت دمج الرسائل المتتابعة جداً |
| `AI_POLL_INTERVAL_MS` | (اختياري) افتراضي 800ms — فحص سريع لطابور الذكاء كشبكة أمان |
| `OUTBOX_POLL_INTERVAL_MS` | (اختياري) افتراضي 500ms — فحص سريع لطابور الإرسال كشبكة أمان |
| `SNAPSHOT_LISTENERS_ENABLED` | (اختياري) افتراضي false — لا تفعّله على الخطط الصغيرة لأنه يستخدم polling مستمر |
| `SNAPSHOT_POLL_INTERVAL_MS` | (اختياري) افتراضي 60000ms عند تفعيل listeners فقط |
| `SUPABASE_CONFIG_CACHE_MS` | (اختياري) افتراضي 60000ms — كاش إعدادات لتقليل قراءات قاعدة البيانات |
| `MESSAGE_SWEEP_ENABLED` | (اختياري) افتراضي false — لا تفعّله إلا للتشخيص لأنه يقرأ محادثات واتساب |
| `EVENT_LOG_ENABLED` | (اختياري) افتراضي false — يقلّل كتابات/قراءات سجل الأحداث |

> مفتاح Groq لكل بوت يُقرأ تلقائياً من Supabase
> (`botSecrets/{botId}.groqApiKey` أو `bots/{botId}.groqApiKey`) فقط، لذا كل بوت
> يعمل بمفتاحه الخاص من اللوحة. لا تضع `GROQ_API_KEY` في Railway؛ إذا حُذف المفتاح من اللوحة يجب أن يتوقف الرد الذكي فوراً.

### 3) النشر
ارفع محتوى مجلد `railway-server` إلى مستودع Railway ثم:
```bash
npm install
npm start
```

### طريقتان للتشغيل
- **الأبسط (خدمة واحدة):** `npm start` — يشغّل الوحدتين معاً في عملية واحدة،
  لكنهما تبقيان منفصلتين في الكود عبر Supabase.
- **خدمتان منفصلتان (فصل كامل):** خدمة بـ `npm run start:bridge` وأخرى بـ
  `npm run start:ai`. الاثنتان تقرآن نفس متغيرات البيئة.

## كيف تتأكد أنه يعمل؟
1. `npm start` → امسح QR من لوحة المتجر (تبويب ربط واتساب).
2. أرسل رسالة تجريبية لرقم البوت.
3. لا تنتظر `remoteSessionSaved` — الحفظ البعيد معطل عمداً. إذا أعدت النشر امسح QR من جديد.
4. راقب بيانات التطبيق: ستظهر الرسالة في `conversations/*/messages`، ثم مهمة في
   `aiQueue`، ثم رد في `outbox` بحالة `sent`.
5. سيصل الرد إلى واتساب بأسرع ما تسمح به واتساب وGroq؛ في التشغيل الافتراضي توجد إشارة فورية، والـ polling الاحتياطي يعمل خلال أقل من ثانية تقريباً.
